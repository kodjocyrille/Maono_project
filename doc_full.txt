MAONO

Document d’Architecture Logicielle

Cible ASP.NET Core / PostgreSQL / Multi-tenant

Version de travail – Avril 2026

Ce document formalise l’architecture recommandée pour Maono Editorial Ops après abandon d’Odoo comme cœur métier. Il décrit l’architecture logique, la structure de solution .NET, la stratégie multi-tenant, les règles transverses et le modèle d’entités cible.

1. Objectif du document
L’objectif de ce document est de fournir un cadre d’implémentation détaillé et cohérent pour la version ASP.NET Core de Maono Editorial Ops. Il sert de référence commune pour l’équipe backend, l’équipe frontend, les parties prenantes produit et les futures décisions d’industrialisation.
La proposition ci-dessous vise un monolithe modulaire, piloté par le domaine métier, avec une exposition HTTP API-first, un stockage PostgreSQL et une stratégie multi-tenant adaptée au produit SaaS visé.
2. Résumé exécutif
Architecture cible : monolithe modulaire ASP.NET Core.
Approche de conception : DDD pragmatique + Application CQRS + MediatR.
Base de données : PostgreSQL.
Multi-tenant recommandé : base partagée, schéma partagé, isolation par WorkspaceId.
Sécurité : ASP.NET Core Identity + JWT Bearer + politiques d’autorisation + Row Level Security ciblée.
Intégrations externes : Odoo en périphérie pour communication/facturation, jamais au cœur du domaine.
Traitements différés : BackgroundService / workers internes pour relances, escalades, publication planifiée et synchronisations.
API : ProblemDetails pour les erreurs, wrapper de succès optionnel dans la couche API uniquement.
3. Principes d’architecture

4. Vue d’ensemble de l’architecture cible
La solution adopte une architecture en quatre couches principales : Domain, Application, Infrastructure et API. Le front Next.js consomme l’API HTTP et les workers internes exécutent les traitements différés. Odoo reste un système périphérique appelé via des ports applicatifs.

6. Règles de dépendance entre projets
Domain ne dépend d’aucun autre projet applicatif.
Application dépend de Domain.
Infrastructure dépend de Domain et Application.
API dépend de Application et Infrastructure.
Aucune dépendance inverse n’est autorisée.
Les intégrations externes sont appelées uniquement via des interfaces exposées par Application et implémentées dans Infrastructure.
7. Règles de conception par couche
7.1 Domain
Contient les règles métier intrinsèques : transitions autorisées, exigences de validation, comportements d’entité, contraintes inter-objets.
Expose des DomainEvents indépendants de MediatR.
Ne contient ni DbContext, ni annotations HTTP, ni logique de sérialisation.
Les agrégats imposent les invariants et protègent l’état.
7.2 Application
Contient les commandes, requêtes et handlers.
Orchestre les transactions et les appels aux repositories.
Déclenche les intégrations via interfaces : IOdooBillingService, INotificationService, IAssetStorageService, etc.
Gère les validations de cas d’usage via FluentValidation.
7.3 Infrastructure
Implémente EF Core et la persistance réelle.
Implémente Identity, JWT, refresh tokens, clients Odoo, email, stockage objet, antivirus éventuel.
Héberge les workers planifiés et les services d’arrière-plan.
7.4 API
Mappe les requêtes HTTP vers les commandes/requêtes applicatives.
Applique l’authentification, l’autorisation, la gestion centralisée des erreurs et la documentation OpenAPI.
Peut envelopper les réponses de succès pour uniformiser le contrat frontend, sans propager ce wrapper dans Application.
8. Flux d’exécution d’une requête
Le client appelle un endpoint HTTP avec un JWT contenant l’identité et le workspace actif.
L’API résout le contexte utilisateur et le contexte workspace.
L’API appelle MediatR avec une commande ou une requête.
Les behaviors exécutent les validations, la journalisation applicative et la transaction si nécessaire.
Le handler orchestre le cas d’usage à partir du domaine et des interfaces abstraites.
Infrastructure exécute la persistance, les appels externes ou le stockage.
Le résultat est transformé en réponse HTTP cohérente.
Les événements métiers ou événements d’intégration sont persistés / propagés selon le scénario.
9. Composants transverses recommandés

10. Stratégie multi-tenant retenue
La stratégie retenue est une base PostgreSQL partagée, un schéma applicatif partagé et une isolation logique par WorkspaceId. Cette approche est la plus adaptée au stade actuel du produit : elle limite la complexité opérationnelle, facilite l’onboarding et reste compatible avec une promotion future de certains clients vers une base dédiée.

10.1 Principes de modélisation multi-tenant
Toutes les entités possédées par un tenant portent un WorkspaceId.
Les utilisateurs sont globaux ; leur appartenance à un workspace passe par WorkspaceMembership.
Le JWT transporte l’identité et le workspace actif.
Toute commande ou requête applicative requiert un contexte tenant explicite.
Les tables globales et les tables tenant-owned sont clairement distinguées.
10.2 Défense en profondeur
Filtrage applicatif : le contexte workspace est résolu en entrée.
Filtrage EF Core : Global Query Filters sur les entités tenant-owned.
Protection PostgreSQL : politiques Row Level Security ciblées sur les tables sensibles.
Audit : chaque trace stocke WorkspaceId.
Stockage fichier : partition logique par workspace dans les clés/buckets.
11. Authentification et autorisation
L’authentification repose sur ASP.NET Core Identity pour la gestion des comptes, mots de passe, rôles, claims et refresh tokens. L’API utilise JWT Bearer comme mécanisme d’authentification principal pour le frontend.

11.1 Permissions
Le système ne doit pas se limiter aux rôles larges. Il doit supporter des permissions fines, par exemple :
campaign.create
campaign.update
calendar.validate
content.assign

workspace.manage_users
12. Politique de réponse API et gestion d’erreurs
L’API peut renvoyer un wrapper de succès unifié pour simplifier l’intégration frontend. Ce wrapper ne doit pas contaminer la couche Application.

13. Intégration avec Odoo
Odoo n’est plus le cœur du produit. Il devient un système externe utilisé pour certaines capacités support : communication, facturation, export administratif ou synchronisation de données si le besoin est confirmé.
Application expose des contrats : IOdooBillingService, IOdooContactSyncService, IOdooCommunicationService.
Infrastructure implémente ces contrats avec HttpClient, mapping, retry, logs et gestion d’erreurs.
Aucune entité métier du domaine ne dépend d’Odoo.
Toutes les synchronisations doivent être auditables et rejouables si nécessaire.
14. Workers et traitements différés
Escalade de deadlines J-7 / J-3 / J-1 / J-0.
Relances d’approbation client après X jours.
Publication planifiée des contenus.
Retry automatique de publication après échec temporaire.
Collecte périodique des métriques.
Synchronisation différée vers Odoo.
Ces traitements seront implémentés via BackgroundService / services hébergés, avec journalisation et idempotence minimale.
15. Journalisation et audit
15.1 Audit métier
Changement de statut d’un contenu.
Approbation interne ou client.
Upload ou restauration d’une version d’asset.
Invitation ou retrait d’un collaborateur sur une mission.
Export d’un document de facturation vers Odoo.
15.2 Logs techniques
Exceptions, appels externes, performances, retries, erreurs réseau, health checks.
Les logs techniques et l’audit métier doivent être séparés conceptuellement et techniquement. Ils n’ont pas le même but ni les mêmes cycles de conservation.

20. Règles d’implémentation EF Core
Ajouter WorkspaceId sur toute entité tenant-owned.
Appliquer des Global Query Filters sur les entités tenant-owned.
Configurer les indexes métiers : WorkspaceId + Status, WorkspaceId + PublicationDate, WorkspaceId + ClientId, etc.
Prévoir des contraintes d’unicité utiles : WorkspaceId + Slug, WorkspaceId + Tag.Name, WorkspaceId + SocialConnection.Platform + ExternalAccountId.
Utiliser des configurations Fluent API plutôt que des annotations éparpillées.
Garder les migrations centralisées dans Infrastructure.
Séparer les projections/reporting si certaines requêtes deviennent trop complexes.

22. Sécurité des données et fichiers
Clés de stockage segmentées par workspace.
URLs signées pour les assets privés.
Historique des téléchargements sensibles si nécessaire.
Rotation des secrets d’intégration.
Antivirus à l’upload pour les documents entrants si le risque le justifie.
Jamais de confiance implicite dans les données externes renvoyées par les plateformes ou par Odoo.

24. Feuille de route technique proposée
1. Créer la solution .NET et les projets Domain / Application / Infrastructure / API.
2. Mettre en place le module Tenancy & Identity.
3. Implémenter le modèle de base : Workspace, Membership, Client, Campaign, CalendarEntry, ContentItem.
4. Ajouter assets, versioning et approbation.
5. Ajouter publication, retry et métriques.
6. Ajouter mode freelance : Mission, Milestones, Deliveries, Billing.
7. Brancher Odoo en périphérie sur les cas d’usage retenus.
8. Durcir la sécurité multi-tenant : query filters + RLS ciblée.
9. Ajouter workers, audit avancé et tests d’intégration.
25. Conclusion
L’architecture proposée répond au besoin réel de Maono : un produit SaaS orienté workflow, piloté par des règles métier fortes, avec une isolation multi-tenant cohérente, une API moderne et une capacité d’évolution maîtrisée. Le choix d’ASP.NET Core comme cœur est pertinent à condition de garder un domaine riche, une application orchestratrice et une infrastructure strictement périphérique.
Le modèle d’entités présenté dans ce document constitue une base suffisamment détaillée pour démarrer la conception EF Core, structurer le backlog backend et aligner les futures API.
